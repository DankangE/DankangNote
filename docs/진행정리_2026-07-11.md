### 2. KAN-4 — 노트 CRUD API ✅ (PR #1, 머지 완료)

**환경**
- 로컬 Postgres: `docker-compose.yml` (postgres:17-alpine, dankangnote DB)
- Prisma 7 + zod + server-only 의존성, `prisma.config.ts`
- 초기 마이그레이션 `20260711000359_init` 생성·적용

**구현** — 폴더 구조 룰(features 수직 슬라이스 + server 전용 레이어) 첫 적용

```
src/
  server/
    db.ts                  # Prisma 싱글턴 (server-only 강제, dev 핫리로드 캐시)
    auth.ts                # requireOrgId() — Clerk 도입 전 임시 org_dev 반환
    services/notes.ts      # 모든 쿼리 orgId 스코프, update/delete는 *Many로 교차 테넌트 차단
  features/notes/
    api/actions.ts         # Server Action: create/update/delete (진입부 requireOrgId + zod)
    api/queries.ts         # 서버 컴포넌트용 fetchNotes/fetchNote
    types.ts               # ActionResult 판별 유니언
```

**핵심 설계 결정**
- 멀티테넌시: 모든 노트 쿼리에 `where: { orgId }` 강제. update/delete는 단건 API 대신 `updateMany`/`deleteMany`로 타 워크스페이스 접근을 원천 차단
- 변이는 Server Action 우선 (Route Handler는 웹훅 등 외부 노출 시에만)
- 모든 외부 입력은 서버에서 zod 검증 후 사용

**런타임 검증 (dev 서버 HTTP 직접 호출)** — 전부 통과
- create/update/delete 정상 동작 및 DB 반영
- zod 검증: 빈 제목·201자 제목·비객체 입력 → 오류 반환
- 멀티테넌시: 타 orgId 노트가 목록 미노출, 해당 id로 delete 거부
- 미존재 id, 중복 삭제 → "노트를 찾을 수 없습니다."

- Node 22 필요 명령: `FNM_DIR=$HOME/.local/share/fnm ~/.local/share/fnm/fnm exec --using 22 -- pnpm <cmd>`
- 로컬 DB: `docker compose up -d` (Claude 세션은 docker 그룹 미적용 — 새 터미널에서 실행)

- GitHub: WSL git은 Windows GCM을 credential helper로 사용, gh CLI는 `~/.local/bin/gh`
