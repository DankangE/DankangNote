# Jira 티켓 + 브랜치 워크플로우

인자로 받은 작업 설명을 바탕으로 Jira 이슈를 만들고, 이슈 키가 포함된 브랜치를 생성해 작업을 시작한다.

## 절차

1. **MCP 연결 확인**: Atlassian MCP 도구가 없거나 인증 오류가 나면, 사용자에게 `/mcp`로 atlassian 서버에 OAuth 로그인하라고 안내하고 중단한다.

2. **프로젝트 키 결정**:
   - 아래 "설정" 섹션에 프로젝트 키가 기록되어 있으면 그걸 사용한다.
   - 없으면 MCP로 접근 가능한 Jira 프로젝트를 조회한다. 하나뿐이면 그걸 쓰고, 여러 개면 사용자에게 고르게 한 뒤 **이 파일의 "설정" 섹션에 키를 기록**해서 다음부터 묻지 않는다.

3. **티켓 생성**:
   - 이슈 타입: 기능이면 Task(또는 Story), 버그면 Bug. 애매하면 Task.
   - Summary: 작업 설명을 한 줄로 (한국어 그대로).
   - Description: 작업 배경·완료 조건을 아는 만큼 채운다. 억지로 지어내지 않는다.
   - 생성 후 이슈 키(예: `KAN-12`)를 확보한다.

4. **브랜치 생성**:
   - main을 최신화(`git fetch origin && git pull --ff-only origin main` — main에 있을 때만)한 뒤 분기한다.
   - 이름 규칙: `<type>/<이슈키>-<영문-슬러그>`
     - type: `feat` | `fix` | `chore` | `refactor`
     - 슬러그는 작업 설명을 짧은 영어 케밥케이스로 (예: `feat/KAN-12-note-editor`)
   - `git checkout -b <브랜치명>` 후 브랜치명을 사용자에게 알린다.

5. **티켓 상태 전환**: 이슈를 "진행 중"(In Progress) 상태로 transition 한다. 해당 상태가 없으면 건너뛰고 그 사실만 알린다.

## 작업 중 규칙 (브랜치가 이슈 키를 포함할 때)

- 커밋 메시지 끝에 이슈 키를 붙인다: `[Feat] 에디터 골격 추가 KAN-12`
- PR 제목에도 이슈 키를 포함한다: `[KAN-12] 노트 에디터`
- PR 생성·머지 후 사용자가 요청하면 이슈를 "완료"(Done)로 transition 한다.
